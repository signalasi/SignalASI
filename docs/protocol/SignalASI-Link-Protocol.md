# SignalASI Link Protocol v1

## Status

SignalASI Link Protocol v1 is a clean-break protocol for communication among clients, servers, devices, and agents. It does not read old pairing state and does not use `signalasichat/android/*`. Every upgraded client must pair again.

Normative terms such as MUST, MUST NOT, SHOULD, and MAY are interpreted as requirements.

## Security Goals

- The relay cannot read message content or impersonate a paired endpoint.
- A leaked MQTT topic is insufficient to decrypt, forge, or replay a message.
- Pairing one client never revokes or replaces another client.
- Revoking one relationship does not rotate unrelated relationships.
- Device identity, transport routing, agent identity, and conversation identity are separate.
- Duplicate MQTT delivery never causes duplicate user-visible messages or task execution.
- Unknown protocol versions, malformed routes, stale claims, and identity mismatches fail closed.

The public relay still observes timing, approximate payload size, and topic activity. TLS is mandatory in production to reduce metadata exposure and active transport interference. Signal encryption remains mandatory even when TLS is enabled.

## Roles and Identifiers

- **Identity ID**: derived from a long-lived Signal identity key. Used for cryptographic trust.
- **Server route ID**: random 128-bit Base64URL value generated by a Desktop, Gateway, or server device.
- **Client route ID**: random 128-bit Base64URL value generated independently by every client installation.
- **Agent ID**: stable identifier within one server, such as `codex` or `hermes`.
- **Conversation ID**: UUID identifying a chat or task thread.
- **Message ID**: UUID generated once by the sender and reused for retries.

Route IDs are opaque capabilities. They MUST NOT contain a user ID, fingerprint, device name, platform, phone number, or email address.

## MQTT Topics

Given server route `S` and client route `C`:

```text
signalasichat/v1/{S}/pair
signalasichat/v1/{S}/{C}/up
signalasichat/v1/{S}/{C}/down
signalasichat/v1/{S}/{C}/control
```

- `pair` accepts only short-lived pairing claims.
- `up` carries encrypted client-to-server traffic.
- `down` carries encrypted server-to-client traffic.
- `control` carries encrypted acknowledgements, revocation, capability, presence, and task events.

All topics use QoS 1 and non-retained business messages. Presence MAY be retained only when it contains no secret or plaintext user data. MQTT client sessions are persistent. Implementations MUST deduplicate by `message_id`.

Development builds default to `ssl://broker.emqx.io:8883`. The EMQX public broker is for development and demonstration only. Production deployments MUST use an authenticated private or managed broker with TLS certificate validation and topic ACLs scoped to the endpoint's routes.

## Pairing

The Desktop exposes `/signalasi/verify`. Its QR payload is:

```json
{
  "type": "signalasi_verify",
  "protocol": "signalasi-link",
  "version": 1,
  "role": "server",
  "desktop_id": "desktop_<fingerprint-prefix>",
  "desktop_name": "Workstation",
  "server_route_id": "<128-bit-route>",
  "pairing_topic": "signalasichat/v1/<server-route>/pair",
  "pairing_token": "<256-bit-one-time-token>",
  "pairing_secret": "<256-bit-base64url-AES-key>",
  "identity_key": "<base64>",
  "identity_key_sha256": "<hex>",
  "signal_bundle": {},
  "created_at": 1780000000
}
```

The client MUST verify the protocol name, exact version, role, route syntax, topic derivation, token presence, identity-key hash, and QR age before asking the user to confirm the displayed server identity.

The client generates its own `client_route_id`. The logical claim is:

```json
{
  "type": "signalasi_pairing_claim",
  "protocol": "signalasi-link",
  "version": 1,
  "pairing_token": "<one-time-token>",
  "server_route_id": "<server-route>",
  "client_route_id": "<client-route>",
  "client_name": "Alice's phone",
  "platform": "android",
  "signal_name": "signalasi:<identity-prefix>",
  "signal_device_id": 1,
  "identity_fingerprint": "<sha256>",
  "signal_bundle": {}
}
```

The logical claim is never published in plaintext. Android encrypts its canonical JSON with AES-256-GCM using `pairing_secret`, a random 96-bit nonce, and authenticated data `signalasi-link|1|{pairing_token}|{server_route_id}`. MQTT carries only:

```json
{
  "type": "signalasi_pairing_ciphertext",
  "protocol": "signalasi-link",
  "version": 1,
  "pairing_token": "<lookup token>",
  "server_route_id": "<server route>",
  "nonce": "<base64url>",
  "ciphertext": "<base64url AES-GCM ciphertext and tag>"
}
```

The pairing secret exists only in the QR and Desktop process memory, expires after ten minutes, and is consumed once. This prevents the MQTT relay from learning the client name, identity fingerprint, or Signal prekey bundle during bootstrap.

The server consumes the token atomically, validates all fields, installs the peer Signal bundle under `signal_name`, and adds a new registry entry. Existing clients are unchanged. Pair confirmation is sent only to the new relationship's `down` topic. Business traffic is rejected until pairing is complete.

## Signal Wire Envelope

MQTT carries a Signal ciphertext envelope:

```json
{
  "protocol": "signalasi-link",
  "version": 1,
  "scheme": "signal",
  "from": "<cryptographic identity name>",
  "to": "<cryptographic identity name>",
  "signal_type": "prekey",
  "message_type": 3,
  "body": "<base64 ciphertext>",
  "time": 1780000000000
}
```

The MQTT topic selects a relationship. The decrypted source identity MUST match the client registered for that relationship. A valid ciphertext on another client's topic MUST be rejected.

## Encrypted Application Envelope

The Signal plaintext is always a structured envelope:

```json
{
  "protocol": "signalasi-link",
  "version": 1,
  "message_id": "12fd6d2b-d95b-4633-91dd-b65e29a61f2d",
  "conversation_id": "41bd43ac-d35a-4dfb-a480-fffbcd84dbdd",
  "source_id": "<client identity>",
  "target_id": "codex",
  "reply_to": "",
  "sent_at": 1780000000000,
  "expires_at": 1780604800000,
  "payload": {
    "type": "text",
    "content": "Inspect the project"
  }
}
```

Required fields are `protocol`, `version`, `message_id`, `source_id`, `target_id`, `sent_at`, `expires_at`, and object `payload`. Receivers reject timestamps more than five minutes in the future and reject expired messages. The default expiry is seven days, allowing persistent offline delivery without weakening replay protection.

Maximum defaults:

- MQTT envelope: 512 KiB.
- Text payload: 128 KiB UTF-8.
- Delivery trace: 32 entries.
- Agent event detail: 16 KiB.
- File metadata: 32 KiB; file bytes use the encrypted file channel.

## Reliability and Idempotency

The sender persists an outbox record before publishing. The receiver persists `message_id` before dispatching side effects. Processing states are:

```text
created -> queued -> published -> received -> decrypted -> accepted -> completed
                                                     \-> failed
```

The receiver sends encrypted acknowledgements containing `message_id`, status, and timestamp. The sender retries with exponential backoff until accepted, expired, cancelled, or the retry limit is reached. A repeated message returns the previous acknowledgement and MUST NOT execute an Agent task twice.

## Capability Discovery

After pairing and whenever capabilities change, the server sends encrypted `capability_manifest` data:

```json
{
  "type": "capability_manifest",
  "manifest_version": 1,
  "server": {"name": "Workstation", "platform": "windows"},
  "agents": [],
  "models": [],
  "tools": [],
  "features": ["tasks", "streaming", "voice", "files"],
  "limits": {"max_parallel_tasks": 4, "max_message_bytes": 524288}
}
```

The UI only exposes capabilities advertised by the selected server. Online transport status and individual Agent readiness are distinct.

## Application Resource Routing

Link pairing and direct cloud-model access are independent. A configured cloud Provider on the client does not require a server relationship and MUST remain callable when no server is paired.

The default Mobile Native Agent policy is:

- With at least one healthy Link v1 server relationship, prefer its advertised Desktop Agents, then fall back to configured direct cloud APIs and other available resources.
- Without a healthy Link v1 server relationship, prefer configured direct cloud APIs and do not attempt unavailable Desktop Agents.
- An explicit user target, local-only requirement, latency mode, cost mode, capability requirement, circuit breaker, or security policy may override this default order.
- A message addressed to a Desktop Agent without a matching paired relationship MUST fail visibly. It MUST NOT fall back to a fixed or public legacy MQTT topic.

## Task Events

Task states are `accepted`, `queued`, `planning`, `running`, `waiting_user`, `waiting_approval`, `paused`, `completed`, `failed`, `cancelled`, and `timed_out`. Every event includes `task_id`, increasing `status_seq`, `conversation_id`, `source_message_id`, timestamps, and a user-safe summary. Private chain-of-thought, secrets, raw environment variables, and unrestricted tool output MUST NOT be transmitted.

## Revocation

Revocation is relationship-scoped. The server first publishes an encrypted `pairing_revoked` event to that client's `control` topic, then marks the registry entry revoked and deletes its Signal session after the delivery window. Future traffic on that route is rejected. Revoking all clients is an explicit separate operation.

A client removing a server performs the symmetric operation. Re-pairing creates a new client route and fresh Signal session; old routes are never reactivated.

## Files and Voice

MQTT carries only metadata or small inline voice payloads within the configured limit. Larger content uses an authenticated encrypted file service. Each object has a random object ID, independent content key, SHA-256 digest, size, MIME type, expiry, and authorization bound to the paired relationship. Temporary objects are deleted after expiry or task cleanup.

## Groups and Peer-to-Peer Extension

Peer relationships use independently generated pair routes and Signal sessions. Groups use Sender Keys, a random group route, and monotonically increasing membership epoch. Adding or removing a member rotates the epoch and sender keys. This extension does not reuse server/client routes.

## Versioning

Unknown major versions are rejected. Additive optional fields are allowed within v1. Removing fields, changing cryptographic meaning, or changing topic semantics requires v2. There is intentionally no compatibility path for fixed `android/send`, `android/recv`, or `android/pc` topics.
